# GDPR Data Flow Summary вҖ” Patent Novelty Checker Agent

## 1. System Overview

The Patent Novelty Checker is a multi-agent system built on LangGraph that assesses the novelty of user-submitted inventions by searching patent and non-patent literature databases. The system:

- Accepts an invention description from the user
- Extracts searchable features via LLM reasoning
- Deploys autonomous subagents to search external patent/literature APIs
- Persists findings to a per-session filesystem workspace
- Generates a novelty assessment report

## 3. Personal Data Inventory

### Data retrieved from external APIs

| Data Field | Source API | In Memory | Persisted to Disk | Sent to LLM |
|------------|-----------|-----------|-------------------|-------------|
| **Inventor names** | Innography | Yes (`Patent.inventors`) | **No** вҖ” filtered out in `search.py` output formatting | **No** |
| **Author names** | Web of Science | Yes (`Article.authors`) | **No** вҖ” filtered out in `search.py` output formatting | **No** |
| **Assignee (org name)** | Innography | Yes (`Patent.assignee`) | Partially (in formatted output, but observed as "N/A" in sessions) | Yes (in search result formatting) |
| Patent titles | Innography, NGSP | Yes | Yes | Yes |
| Abstracts | Innography, WOS, NGSP | Yes | Yes (truncated to 500 chars) | Yes |
| Publication numbers | All APIs | Yes | Yes | Yes |
| Priority dates | Innography | Yes | Yes | Yes |

### Data provided by the user

| Data Field | Persisted | Location |
|------------|-----------|----------|
| Invention description | Yes | `scope.md`, `traces/*_messages.json` |
| Feature specifications | Yes | `features.md`, `traces/*_messages.json` |

### Data generated by the system

| Data Field | Contains Personal Data | Location |
|------------|----------------------|----------|
| Search queries (tool args) | No (LLM-generated keywords) | `findings/auto/*.json`, `telemetry.json` |
| Tool call metrics | No (timestamps, durations) | `telemetry.json` |
| Patent funnel statistics | No (aggregate counts) | `patent_statistics.json` |
| Final report | No (references by publication number) | `final_report.md` |

## 4. External API Data Sources

| API | Provider | Data Retrieved | Authentication | Data Direction |
|-----|----------|---------------|----------------|---------------|
| Innography | Clarivate | Patent records (titles, abstracts, inventors, assignees, claims) | API key + credentials | **Read-only** вҖ” no user data uploaded |
| Web of Science | Clarivate | Article records (titles, abstracts, authors, journals, DOIs) | API key (Bearer) | **Read-only** вҖ” no user data uploaded |
| NGSP | Clarivate | Semantic patent matches (titles, similarity scores) | API key | **Read-only** вҖ” no user data uploaded |
| Azure OpenAI | Microsoft | LLM inference (reasoning, analysis) | API key | **Bi-directional** вҖ” invention text + patent metadata sent; reasoning returned |

**Note:** All search APIs receive only LLM-generated keyword queries, not raw user input. No user data or personal data is uploaded to Innography, WOS, or NGSP.

## 5. Data Retention and Cleanup

- **Automatic cleanup:** `cleanup_old_sessions(max_age_hours=24, max_sessions=50)` runs at agent startup
- **Session isolation:** Each session gets its own directory under `sessions/{thread_id}/` via `ThreadAwareBackendFactory`
- **No cross-session data sharing:** Sessions do not read from or write to other sessions
- **No long-term storage:** No database, no cloud storage вҖ” filesystem only
- **No user profiles:** No user accounts, no persistent user identity across sessions

## 6. Design Safeguards in Place

| Safeguard | Implementation | File |
|-----------|---------------|------|
| Inventor names filtered from output | `search.py` formats patent results without `inventors` field | `src/tools/search.py` |
| Author names filtered from output | `search.py` formats NPL results without `authors` field | `src/tools/search.py` |
| Per-session isolation | `ThreadAwareBackendFactory` creates separate filesystem workspace per thread | `src/novelty_checker/backend_factory.py` |
| 24-hour auto-deletion | `cleanup_old_sessions()` removes expired session directories | `src/novelty_checker/deep_agent.py` |
| Normalized persistence | `FindingsPersistenceMiddleware` extracts only structured fields (pub number, title, date, abstract) | `src/novelty_checker/middleware/findings.py` |
| Local-only telemetry | No remote telemetry services (no Datadog, Sentry, etc.) | `src/novelty_checker/observability/telemetry.py` |
| Content filtering | `ContentFilterMiddleware` catches Azure content policy violations gracefully | `src/novelty_checker/middleware/` |

## 7. Known Gaps and Recommendations

### Gaps

1. **No defense-in-depth on personal data filtering.** Inventor/author names are filtered at the tool output layer (`search.py`), not at the persistence layer. If the output format changes to include these fields, `FindingsPersistenceMiddleware` would silently persist them.

2. **Invention descriptions sent to Azure OpenAI.** User-provided invention text is sent to the LLM for reasoning. GDPR compliance for this data flow depends on the organization's Data Processing Agreement (DPA) with Microsoft/Azure вҖ” this is a deployment/legal concern, not a design issue.

3. **No manual deletion mechanism.** Users cannot request immediate deletion of their session data. The 24-hour auto-cleanup is the only retention mechanism.

4. **Message traces contain full conversation history.** `traces/*_messages.json` stores complete LLM message history, including invention descriptions and all tool call results.

5. **Assignee names (organization/company) are included in LLM context.** While not personal data in most cases, company names from Innography are passed to the LLM in patent search results.

### Recommendations (design-level)

1. **Add persistence-layer filtering:** Strip any personal data fields (inventors, authors) in `FindingsPersistenceMiddleware._extract_references_from_result()` as a second line of defense.

2. **Document legal basis:** The processing of patent inventor names (even transiently in memory) obtained via Innography's licensed API requires a documented legal basis under GDPR Article 6(1)(f) (legitimate interest) and consideration of Article 14 (indirect collection).

3. **Review Azure OpenAI DPA:** Ensure the organization's Azure agreement covers the processing of invention descriptions sent to the LLM. This is a deployment/procurement concern.
